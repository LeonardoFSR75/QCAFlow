1. Arquitetura do Sistema

VISÃO GERAL

A ferramenta é implementada como um único arquivo HTML5 (single-page application). A partir da v4.1, a biblioteca SheetJS (usada apenas para importar Excel .xlsx) é embutida no próprio arquivo, e na v5.0 a lógica de calibração, necessidade, tabela verdade e dashboard foram estendidas para suportar calibração linear, contradições de caso e negação do desfecho. A ferramenta funciona 100% offline, residindo toda a lógica em JavaScript puro (ES6+) e persistindo os dados via localStorage.

3. Algoritmos Implementados

ALGORITMO 1

3.1 Geração da Tabela Verdade

Para N condições, são geradas todas as 2ᴺ combinações binárias possíveis em ordem decrescente (de 111...1 a 000...0).

function generateCombinations(n) {
  const total = Math.pow(2, n);
  const combos = [];
  for (let i = total - 1; i >= 0; i--) {
    const bits = [];
    for (let b = n - 1; b >= 0; b--) bits.push((i >> b) & 1);
    combos.push(bits);
  }
  return combos;
}

Complexidade: O(2ᴺ × M), onde M é o número de casos. Para N=5: 32 combinações; para N=8: 256.

ALGORITMO 2 v4.1

3.2 Calibração — Método Direto (Ragin, 2008)

Transforma valores brutos em escores de pertencimento fuzzy [0,1] a partir de três âncoras qualitativas: pertencimento pleno (≈0,95), ponto de cruzamento (0,50) e não-pertencimento pleno (≈0,05), via transformação log-odds.

function calibValue(x, full, cross, non) {
  const lo = x >= cross ? 3*(x-cross)/(full-cross)   // ramo superior
                        : 3*(x-cross)/(cross-non);   // ramo inferior
  const m = 1 / (1 + Math.exp(-lo));                 // logística → [0,1]
  return Math.round(m*1000)/1000;
}
// Âncoras sugeridas automaticamente pelos percentis 95/50/5 (suggestAnchors);
// escores exatamente = 0,500 são ajustados para 0,501 e registrados em calibInfo.adjusted.
// undoCalibration() restaura rawBackup (dados brutos pré-calibração).
Boa prática: as âncoras devem ser justificadas por teoria ou conhecimento substantivo, não apenas pela distribuição dos dados.
ALGORITMO 3

3.3 Consistência, PRI e Cobertura (fuzzy)

As métricas usam somas de pertencimento mínimo sobre todos os casos (M = pertencimento do caso à configuração/termo; Y = pertencimento ao desfecho). Reduzem-se às contagens clássicas com dados crisp.

MétricaFórmulaInterpretação
Consistência de SuficiênciaΣ min(M, Y) / Σ MGrau em que M ⊆ Y
Consistência PRI v4.1(Σmin(M,Y) − Σmin(M,Y,1−Y)) / (ΣM − Σmin(M,Y,1−Y))Reduz inconsistência por pertencimento simultâneo a Y e ~Y
Cobertura BrutaΣ min(M, Y) / Σ YFração do desfecho explicada pela config.
Cobertura Únicacob(solução) − cob(solução sem o termo)Fração explicada exclusivamente por este termo

Um limiar de frequência (limiarFreq) roteia linhas com poucos casos aos remainders; o % de casos retidos é exibido. O alerta de PRI dispara em linhas suficientes com PRI < limiarPRI ou distância consist−PRI > 0,25.

ALGORITMO 4

3.4 Minimização Booleana — Quine-McCluskey e as Três Soluções

O algoritmo combina iterativamente pares de implicantes que diferem em exatamente uma posição, gerando termos com "don't care". A seleção de cobertura (implicantes primos essenciais + guloso) garante que apenas as linhas observadas suficientes precisem ser cobertas.

// Estrutura interna de um implicante
{
  bits:   number[],  // Valores das condições (0 ou 1)
  dc:     number[],  // Máscara de don't-care (1 = irrelevante)
  covers: string[],  // Mintermos cobertos ex: ["110", "111"]
  core:   boolean[]  // (só na intermediária) nuclear × periférica  [v4.1]
}

A Análise-Padrão (computeSolutions) deriva três soluções [Ragin, 2008]:

SoluçãoRemainders usadosUso
Complexanenhumconservadora — apenas evidência observada
Parcimoniosatodos que simplificamidentifica condições nucleares
Intermediáriasó contrafactuais fáceis (dirExpect)padrão de relato

Núcleo × periferia (Fiss, 2011): um literal da intermediária é nuclear quando pertence a um termo parcimonioso nele aninhado; caso contrário, é periférico.

Debate Causal Configuracional (Soluções): Embora a abordagem clássica de Ragin (2008) sugira o uso da solução intermediária para relato em virtude de sua coerência com as expectativas teóricas do pesquisador, estudos e simulações com busca reversa de causalidade (Baumgartner & Thiem, 2017) demonstram que as soluções intermediária e complexa geram redundâncias causais incorretas e falsos positivos sob condições estritas de descoberta causal. Baumgartner e Thiem defendem que apenas a solução parcimoniosa recupera corretamente o grafo causal gerador dos dados. Em contrapartida, os defensores da QCA tradicional apontam o risco de a parcimônia incorporar suposições contrafactuais teoricamente absurdas ou biologicamente implausíveis. O QCAFlow aborda essa controvérsia ao disponibilizar as três soluções lado a lado de forma clara e interativa, permitindo que o pesquisador visualize as implicações de cada premissa de minimização e realize decisões de modelagem transparentes e documentadas.
Abordagens Alternativas (CNA e Otimização): Como alternativa à minimização booleana tradicional baseada em tabelas verdade e counterfactuals, a Análise de Coincidência (Coincidence Analysis — CNA, Baumgartner & Ambühl, 2020) modela relações causais sem o uso de logical remainders, permitindo desvendar estruturas complexas como causas comuns (efeitos colaterais) e cadeias causais (onde uma variável é desfecho intermediário e causa subsequente). Além disso, o protocolo ConCovOpt (Baumgartner & Ambühl, 2021) provê um algoritmo sistemático de otimização de consistência e cobertura que auxilia na identificação da fronteira de eficiência empírica dos modelos, eliminando arbitrariedades na escolha de limiares.

Complexidade: converge em 2–3 iterações para N ≤ 8 na prática.

ALGORITMO 5

3.5 Análise de Necessidade com RoN

Para cada condição Cᵢ e sua negação ~Cᵢ são calculadas consistência e cobertura de necessidade (fuzzy) e a Relevance of Necessity (RoN; Schneider & Wagemann, 2012), que distingue necessidade substantiva de necessidade trivial (condição quase constante na amostra).

// Fuzzy, para X = pertencimento à condição (ou 1−X para a negação):
consistNec = Σ min(X, Y) / Σ Y;
cobNec     = Σ min(X, Y) / Σ X;
RoN        = Σ (1 − X) / Σ (1 − min(X, Y));   // ~1 = relevante; baixo = trivial
isNec      = consistNec >= 0.90;
trivial    = isNec && RoN < 0.5;              // sinaliza necessidade trivial
ALGORITMO 6 v4.1

3.6 Ausência do Desfecho (~Y) e Testes de Robustez

Assimetria causal: computeNegSolution() deriva a solução conservadora de suficiência para a ausência do desfecho (~Y), com os mesmos limiares, e sinaliza qualquer linha codificada como suficiente para Y e ~Y simultaneamente.

Robustez (runRobustness()): reexecuta a análise (a) variando o limiar de consistência em 0,70–0,90 e (b), se houver calibração aplicada, deslocando o ponto de cruzamento das âncoras em ±5% do intervalo pleno–não, reportando se a solução intermediária permanece idêntica.

5.5 Formato Excel (.xlsx)

Importado via importExcel(event) usando SheetJS. A ferramenta lê a primeira aba da planilha. A primeira linha deve conter os cabeçalhos na ordem: Caso, Cond_1, ..., Cond_N, Resultado. As colunas são detectadas automaticamente por posição.

v4.1: a biblioteca SheetJS é embutida no HTML — a importação Excel funciona 100% offline, sem qualquer dependência de CDN ou conexão à internet.

5.6 HTML Auto-contido

Exportado via exportSelfContained(). Injeta o estado atual diretamente no bloco init() do HTML, substituindo os dados de amostra pelos dados reais. O arquivo resultante é completamente funcional sem qualquer dependência externa.

5.7 Histórico JSON

Exportado via Exportar Histórico (JSON). Contém array de todas as sessões salvas. Pode ser importado em outra instância da ferramenta para transferência do histórico entre dispositivos.

7. Implantação e Distribuição

DEPLOYMENT

7.1 Uso Local (recomendado para pesquisa)

  1. Copiar o arquivo QCAFlow.html para qualquer diretório.
  2. Abrir com qualquer navegador moderno (duplo clique ou Abrir com...).
  3. Funciona 100% offline — incluindo a importação de Excel (SheetJS embutido). Nenhuma conexão à internet é necessária.
  4. O histórico é salvo automaticamente no localStorage sob a chave qcaflow_historico.
Atenção: O localStorage é específico por origem (browser + perfil). Para transferir histórico entre computadores, use Exportar Histórico (JSON) na aba Histórico.

7.2 Distribuição entre Pesquisadores

  • Opção A — Arquivo .qcaflow: exporte o projeto via Exportar Projeto (.qcaflow) e compartilhe o arquivo. O destinatário importa na sua instância da ferramenta.
  • Opção B — HTML auto-contido: exporte via Exportar HTML Completo. O arquivo resultante contém os dados embutidos e pode ser aberto diretamente por qualquer pessoa.
  • Opção C — Email/repositório: anexe o .html ou .qcaflow em e-mail, Google Drive, GitHub, OSF ou Zenodo.

7.3 Hospedagem Web (opcional)

O arquivo pode ser publicado em qualquer servidor HTTP estático (GitHub Pages, Netlify, etc.) simplesmente fazendo upload do .html. Não requer backend, banco de dados ou configuração especial.

# Exemplo — GitHub Pages
git init
git add QCAFlow.html
git commit -m "Publicar ferramenta fsQCA"
git remote add origin https://github.com/usuario/fsqca-tool.git
git push -u origin main
# Ativar GitHub Pages em Settings > Pages > Branch: main

9. Histórico de Versões

CHANGELOG

v5.0 — Versão Atual ATUAL

  • NOVO Calibração Linear por Partes — alternativa metodológica à transição logística tradicional (Nikou et al., 2024)
  • NOVO Contradições Fuzzy de Caso — identificação automática de inconsistências usando consistência individual ($ObsConsist_c$, Rubinson, 2013) com limiar de proporção
  • NOVO Override Manual de Resultado — possibilidade de alterar manualmente o resultado na tabela verdade para Suficiente (1), Insuficiente (0) ou Contradição (C)
  • NOVO Necessidade Lado a Lado ($Y$ e $\sim Y$) — tabelas separadas para a presença e ausência do desfecho com cálculo de RoN
  • NOVO Dashboard com KPI de Contradições — card indicador de contradições e status badges dinâmicos na tabela verdade
